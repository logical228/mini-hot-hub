## 🧭 一、产品概述

**Mini Hot Hub（今日热搜）** 是一个多平台热搜聚合门户。首页以卡片网格展示微博、知乎、B 站等主流平台热榜，用户零登录即可浏览全网热点，一键跳转原文。

**核心价值主张**：*60 秒内掌握中文互联网正在发生的事。*

## 🏗️ 三、核心功能

### 3.1 MVP（本期必须交付）

#### 🔹 F1：首页卡片网格

| 要素 | 规格 |
|---|---|
| 卡片数量 | ≥ 3 张（微博热搜 / 知乎热榜 / B 站热搜） |
| 卡片头部 | 平台名称 + 榜单名称 + 更新时间 |
| 列表内容 | 每条含排名、标题、热度值（可选）、跳转链接 |
| 排名强调 | TOP 1-3 视觉放大 / 高亮（火焰图标 🔥） |
| 空态处理 | 数据为空时显示「暂无数据」占位 |

#### 🔹 F2：数据接口层（自建后端）

| 接口 | 方法 | 说明 |
|---|---|---|
| `/api/hot/:source` | GET | 按平台获取热榜（source = weibo \| zhihu \| bilibili） |
| `/api/hot` | GET | 一次性返回全部平台聚合数据 |
| 响应格式 | JSON | 统一遵循 `HotPlatform` 结构（详见 Tech Design） |

#### 🔹 F3：智能缓存

- 每个平台独立缓存，TTL 可配置（默认 600 秒）
- 页面展示相对时间文案：「更新于 3 分钟前」
- 缓存命中时直接返回，不触发上游请求

#### 🔹 F4：异常隔离

- **单平台失败不影响其他卡片**：采用 `error-boundary` 思路，每张卡片独立数据通道
- 失败卡片展示友好提示：「该榜单暂时无法加载，请稍后再试」
- 全局 loading 态：首次加载展示骨架屏（Skeleton）

#### 🔹 F5：公网部署

- GitHub 托管代码，README 含部署指南
- 前端部署至 Vercel（自动 HTTPS）
- 后端部署至 Railway / Render（免费额度）
- 生产环境 CORS 白名单配置

## 🎨 四、界面设计要求

### 4.1 设计风格

- **关键词**：极简、信息密度适中、卡片式、微动效
- **参考标杆**：今日热榜（布局结构）、Linear（配色克制）、Vercel Dashboard（卡片精致感）
- **差异化**：比今日热榜更清爽，比 Hacker News 更现代

### 4.2 响应式规范

| 断点 | 布局 | 卡片列数 |
|---|---|---|
| ≥ 1024px | 桌面 | 3 列 |
| 768-1023px | 平板 | 2 列 |
| < 768px | 手机 | 1 列 |

### 4.3 视觉细节

- **平台色系**：微博 #E6162D 🔴、知乎 #0084FF 🔵、B站 #FB7299 🩷
- **卡片阴影**：`box-shadow: 0 2px 12px rgba(0,0,0,0.08)`，hover 微上浮 2px
- **排名徽章**：TOP1 金色、TOP2 银色、TOP3 铜色，字号 1.2x
- **页脚**：居中排列，小字号灰色，含「学习项目 · 非商用声明」+ 数据来源链接

## 🛠️ 五、技术栈概览

| 层级 | 技术选型 | 选型理由 |
|---|---|---|
| 前端框架 | React 18 + TypeScript | 生态成熟、类型安全、Vibe Coding 首选 |
| 构建工具 | Vite | 极速 HMR、零配置上手 |
| 样式方案 | CSS Modules / Tailwind CSS | 作用域隔离 / 原子类快速排版 |
| 后端框架 | Node.js + Express | 轻量、JS 全栈统一语言 |
| 缓存层 | 内存 Map（可升级 Redis） | MVP 阶段够用，后期可横向扩展 |
| 部署平台 | Vercel（前端）+ Railway（后端） | 免费额度、自动 CI/CD、原生 HTTPS |

## 💡 七、打卡思考 & AI 优化建议

### Q：AI 对你的 PRD 提出了哪些优化建议？（≥3 条）

> 以下为模拟 AI Code Review 后给出的典型优化建议：

**建议 1：补充异常态的完整用户旅程**
> PRD 提到了「单平台失败显示友好提示」，但未覆盖「全部平台同时失败」的极端场景。建议补充：全平台故障时展示一个全局 Fallback 页，含「重试」按钮和预计恢复时间说明。

**建议 2：缓存失效策略需要更细粒度**
> 仅设置 TTL 不够，建议增加「缓存预热」机制——后端启动时主动 fetch 一次各平台数据，避免首个用户请求时冷启动延迟。

**建议 3：补充 SEO 与社交分享优化**
> 作为可公开访问的页面，建议在 PRD 中增加 `<head>` 元数据要求（title / description / OG 标签），让分享到社交媒体时展示正确的标题和描述。

**建议 4（额外 bonus）：数据来源合规性**
> 建议在页脚不仅标注「非商用」，还增加各平台数据来源的外链跳转（如微博热搜榜官方页），降低合规风险。

**建议 5（额外 bonus）：增加 Loading 骨架屏的具体规范**
> PRD 提到了 Skeleton，建议明确骨架屏的动画节奏（如 shimmer 效果 1.5s 循环）、占位行数（每张卡片 8-10 行），让前端实现有章可循。
